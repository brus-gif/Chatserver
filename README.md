# Chatserver
可以工作在nginx tcp负载均衡环境中的集群聊天服务器和客户端源码，基于muduo，mysql，redis实现，正在断改进中

项目目标：
开发面向海量用户的高并发在线聊天服务器，支撑大规模实时通讯场景。
搭建高承载即时通讯服务，聚焦高用户量、低延迟的在线聊天能力实现。

当前项目描述:
v1.0：可以工作在nginx tcp负载均衡环境中的集群聊天服务器和客户端源码，基于muduo，mysql，redis实现
一.项目介绍
src:存放所有代码的源文件(server目录下的是服务端代码，client目录下的是客户端代码)
include:存放所有代码头文件
build:存放编译连接过程中产生的所有.o文件等副产物
bin:存放最终生成的可执行文件
1.项目服务端代码总体分四层框架:
网络层:chatserver.cpp与chatserver.hpp
基于muduo实现，负责网络模块。处理客户端的连接请求和事件处理，对于具体的业务请求，只做筛选，不做处理。
业务层:chatservice.cpp与chatservice.hpp
封装了各个数据映射层的对象来实现业务功能。使用了懒汉式单例模式，封装存储各个业务的回调函数的容器_msghandlerMap，并注册各个回调函数，通过网络层筛选的请求调用对应的回调函数处理具体业务。但它只负责调用各个映射层对象的函数来实现处理各个业务数据，只做逻辑调用处理，不接触具体的数据处理。
数据处理层：分为redis，Model，db四个模块，对应项目中的目录
各个模块封装对应的底层数据映射的对象，类内部封装对外暴露的功能性接口，用来提供业务层处理对应的业务逻辑。他们只负责各自的对应模块的接口处理封装，实现自己模块的功能性接口。而他们对外暴露的函数内部可以处理数据库与redis中的对应数据，也可以处理数据映射层对象的数据。
数据映射层:user.hpp,group.hpp,public.hpp
负责建立数据映射，考虑到数据库查询修改的性能消耗，数据库与内存的io交互。设计了数据映射层来临时性存储数据与中转数据，方便后续业务扩展，也可以有效避免频繁的数据库访问
2.项目的客户端分三层框架:
公共层:clientserver.hpp与clientserver.cpp
处理客户端的基础业务，负责校验客户信息，提供给客户登陆前的基础功能。所有封装的函数只接收用户的基本指令和数据，内部封装具体的数据处理与发送逻辑
用户层:clientChatModel.hpp与clientChatModel.cpp
处理客户端的具体功能接口的封装，将用户登陆后的详细功能接口封装，处理用户的详细请求，提供主要的服务。
数据映射层：与服务端代码一样。
3.使用了nginx tcp负载均衡，以权重配比实现请求分发。redis配置一个用户一个频道，用户登录服务器订阅该用户的专属频道退出时服务器取消订阅的频道

3.副本问题：
vscode远程编译可能连接不稳定，vscode会自动把当前修改的文件生成快照/副本，并在文件后面附上数字，表示第几个副本。但是副本不等于源文件，在vscode编译链接时默认会编译链接原文件，而不是副本。如果副本没更新覆盖源文件，就会报错。
当你写好代码点击编译时，CMake 为了追求极致的速度，会把**中间生成的配置文件、头文件路径、还有上一次编译失败的残渣全都死死地记录在一个叫 CMakeCache.txt（CMake缓存）的文件里。下一次编译时，它如果发现这部分没变，就直接偷懒用旧的。
（1）第一步：CMake: Clean Reconfigure
这一步主要针对的是 build 目录下的 Makefile 和之前编译生成的 .o 目标对象文件的清理
（2）第二步：CMake: Clean Delete Cache and Reconfigure
粉碎缓存，它会把 build 目录下的 CMakeCache.txt（缓存文件） 直接物理粉碎删除，然后重新执行一遍 cmake ..
（3）第三步：点击 Build 按钮
从零构建（纯净编译），这时候底部的 CMake 插件会调用 make 或者 g++ 编译器，开始真正的编译。
4.在构建连接容器时，将Tcpconnectionptr存为了Tcpconnection，之后插入连接使用了emplace导致错误延展。emplace会为了达到类型匹配去寻找对应构造函数，导致我传入的的Tcpconnectionptr也会被转为Tcpconnection，编译器不报错，但逻辑错误。在对库使用不熟悉，优先使用insert等直接匹配函数，防止未知错误扩展。
5.没有异常保护装置：
要在onMessage的业务通信逻辑中的json通信位置加上try catch，用来捕捉异常信号，或者用signal并加上异常回调函数。避免客户操作不规范，导致服务器代码运行异常，发出的异常信号直接把整个服务器进程打挂掉
6.绝对！绝对！绝对！不要因为锁的粒度问题而把unordered_map中存储的TcpconnectionPtr进行随意赋值。因为转移走的是一个指针！智能指针会直接把对应位置的键值对的值掏空掉，变成nullptr，后续访问就是空指针，直接段错误报Segmentation fault (core dumped)。这里TcpconnectionPtr是个强智能指针，注意赋值一定不能用move
7.在进行数据库有关的操作时，应该先判断是否成功连接数据库if(mysql.connect())。
获取离线消息时，在得到MYSQL_ROW*结果集后，应该先判断拿到的结果集是不是
nullptr，因为在每次登录前都会有获取离线消息的操作，不先判断nullptr，直接进行mysql_fetch_row查询，就可能会发生段错误Segmentation fault (core dumped)。
8.关于在每个数据库映射模块函数中都创建MySQL实例而不统一封装在一个数据映射模块类中的原因：
可以把一个 MySQL 实例（句柄）封装进类里让所有函数共用。但是，不这么做是因为在真实的开发场景（尤其是多线程或 Web 后端）中，这样做会导致非常严重的后果。
方案 B（一个类实例内部只有一个 MYSQL* 句柄），最大的死穴在于：MySQL 官方的 C API 句柄是非线程安全的，且不支持并发交织请求。）致命的多线程冲突（Crash 或数据串线）
(1)程序有多个线程（比如 Web 服务器同时处理多个用户的请求），它们同时调用这个映射层类的不同函数：
线程 1 正在执行 SELECT 并准备用 mysql_fetch_row 一行行读数据。此时 线程 2 进来了，调用了 UPDATE 函数，往同一个 MYSQL* 句柄里发送了新指令。这时，MySQL 就会直接报错（通常是 Commands out of sync; you can't run this command now），甚至直接导致程序崩溃（Segment Fault）。因为一个基础的 MYSQL 链接在同一时刻只能做一件事。
(2) 无法同时处理多个结果集
即使是单线程，方案 B 也会遇到麻烦。比如你写了这样的伪代码：
// 伪代码：查询所有用户，并根据用户的角色去查权限
auto users = db.getAllUsers(); // 内部执行了 SELECT，拿到了结果集
for(auto& user : users) {
    auto target = db.getRows(); // 如果这里共用一个句柄，上一个查询还没结束，这里就会直接报错！
}
(3)为什么要在每个函数里都“新建”实例？
方案 A（每个函数独立创建、连接、关闭），虽然看起来“效率低”，但它有几个无法替代的优点：
天然的线程安全：每个函数内部的 MYSQL* 实例都是局部变量，生存在各自线程的栈上，互不干扰。绝对不会发生“线程 A 的数据被线程 B 串掉”的情况。
逻辑极度简单：用完就释放（mysql_close），不用担心连接断开、状态污染、或者长时间不用被 MySQL 服务端强制断开（Wait Timeout）的问题。
对于教学项目，或者并发量极低的工具型项目，方案 A 是最稳妥、最不容易写出 Bug 的做法。
(4)真正的企业级解决方案：数据库连接池（Connection Pool）
既然方案 A 频繁创建/断开连接太慢（每次都有 TCP 三次握手和 MySQL 认证开销），方案 B 共用一个又会崩溃，那企业里怎么做？
答案是:数据库连接池。
它的工作原理非常巧妙：
 1. 程序启动时，在内存里一口气创建好 10 个 MYSQL 实例，放到一个“池子”（队列）里。
 2. 当你的映射层函数需要操作数据库时，不去新建，而是去池子里“借”一个空闲的实例。
 3. 拿到实例后执行增删改查。因为这个实例此时被你独占，所以绝对安全。
 4. 函数执行完毕后，不销毁它，而是把它“还”给池子，让其他函数或线程继续用
9.在设置异常信号的捕获函数和回调时，如果使用的是sigacton，就要注意他会屏蔽掉当前的信号。想要不让他阻塞当前信号就要在sa_flags设置为SA_NODEFER，注意，一定要在回调函数中添加exit退出进程的操作，因为回调操作将会代替默认的系统exit操作（如果想要退出的话）。

14.atoi函数：
atoi 核心功能（标准 C 库：int atoi(const char *nptr);）
1. 完整功能逻辑
跳过字符串最前面所有空白字符（空格、\t、\n）；
识别可选正负符号 + / -；
连续读取后续十进制数字字符，碰到第一个非数字字符立刻终止解析；
把读到的数字转为 int 整型返回；
首字符无有效数字（字母、符号、空串），固定返回 0；
数值溢出超出 int 值域：未定义行为（UB，结果不确定）
这里，由于在第一个字符不为数字时会返回0，所以如果数据库对应表的对应字段不是自增属性（自增一般从1开始以后的），就要加上check(!=0)的约束。

重大逻辑错误：
recvtaskhandler客户端线程回调函数是死循环的，在一开始我考虑到切换用户问题，采用了线程对象，但是却忽略了移动构造会把正在活跃的线程掏空替换，导致aborted错误。之后采用了线程复用，因为切换用户并不断开与服务器的连接，通信套接字不会改变，可以让切换的用户继续服用原来的子线程（recvtaskhandler）接收。但是客户端子线程和客户端主线程共用一个通信套接字，这样做会导致在客户端登陆时，本该由可客户端主线程接收的服务器登录响应信息被子线程recv（recvtaskhandler）接收了。导致了主线程不知道登录是否成功而被recv阻塞，整个业务循环阻塞。
18.
 redisCommand 的 %s 格式化，只能接收 const char*，直接用 %s 时，只能传 const char* 类型的字符串，不能直接传 std::string 对象。
因为 redisCommand 底层是用 printf 风格的格式化实现的，%s 要求传入的是以 \0 结尾的 C 风格字符串指针。
直接传 std::string 对象，会把它的内存地址（指针值）当成字符串解析，直接导致乱码或崩溃。
19.一个用户一个频道（每个用户订阅自己的专属频道）
这是理论上最 "高效" 的方案，但实际上完全不可行。
工作流程
用户登录时，服务器订阅这个用户的专属频道 user:123
用户退出时，服务器取消订阅这个频道
任何人要发消息给用户 123，直接发布到 user:123 频道
只有用户 123 所在的服务器会收到这条消息
性能数据（100 台服务器集群，100 万在线用户）
订阅总数：100 万个（每个在线用户 1 个）
每条跨节点消息产生的 Redis 消息数：1 条
有效消息率：100%
Redis 网络带宽消耗：每条消息 × 1 倍
为什么完全不可行？
Redis 订阅数上限：单台 Redis 最多支持约 10 万个并发订阅，100 万用户直接把 Redis 搞挂
订阅 / 取消订阅开销极大：用户登录 / 退出时，都要和 Redis 交互一次，100 万用户同时上线会把 Redis 打垮
连接数爆炸：每个订阅都占用一个 Redis 连接，100 万订阅需要 100 万个 Redis 连接
故障转移复杂：用户重连到其他服务器时，需要取消旧服务器的订阅，建立新服务器的订阅，很容易出现重复订阅或者订阅丢失
